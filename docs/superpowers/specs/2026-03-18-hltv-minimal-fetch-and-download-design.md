# HLTV 最小抓取与 Demo 下载设计

- 日期：2026-03-18
- 项目：CS2DemoPlayer
- 状态：已确认设计，待进入实现计划

## 1. 背景

当前项目的下一阶段目标之一是接入 HLTV 或可替代来源的比赛与 demo 元数据，为后续的 demo 库补齐外部比赛上下文。

本轮不追求完整数据接入，只追求一条可验证的最小链路：

1. 获取最近比赛
2. 进入某场比赛详情
3. 成功下载对应 demo 到本地

已有事实：

- 当前环境下，直接对 `hltv.org` 发起普通 HTTP 请求会返回 `403`
- 原始 `hltv` npm 包最新版本为 `3.5.0`，最后发布时间为 `2023-09-20`
- `@beermonster/hltv` 是仍在维护的 fork，最近发布时间为 `2026-02-04`
- 原始 `HLTV` 库支持自定义 `loadPage`，说明其可作为解析层复用，但不能假设默认 HTTP 抓取在当前环境可用

## 2. 目标与非目标

### 目标

- 打通一条“最近比赛 -> 比赛页 -> demo 下载”的最小可行路径
- 为后续接入 UI、数据库和批量同步提供真实技术依据
- 优先验证 HLTV 是否能在浏览器自动化路径下稳定拿到 demo

### 非目标

- 不在这轮接入主界面或 demo 库 UI
- 不在这轮设计完整比赛数据模型
- 不在这轮做批量同步、定时抓取或账号体系
- 不在这轮接入 Liquipedia、PandaScore、GRID 等商业或付费数据源

## 3. 方案比较

### 方案 A：Playwright 直接抓 HLTV 页面并直接下载 demo

做法：

- 用 Playwright 打开 HLTV 最近比赛或结果页
- 找到最近且带 demo 的比赛
- 进入比赛页后直接触发 demo 下载
- 同时从页面提取最小元数据

优点：

- 最短路径，最符合当前目标
- 不依赖第三方非官方解析器是否跟上页面变更
- 可以直接验证真实下载链路

缺点：

- 选择器对页面结构变更敏感
- 需要处理 Cloudflare、跳转和下载等待

### 方案 B：Playwright 获取页面 HTML，再交给 `HLTV` parser family 解析

做法：

- 由 Playwright 负责拿到页面内容
- 用 `@beermonster/hltv` 或原 `hltv` 的解析能力提取比赛数据

优点：

- 结构化解析可能更省代码
- 后续扩展到更多比赛字段时可复用现有 parser

缺点：

- 比方案 A 多一层耦合
- parser 仍受 HLTV 页面结构变化影响
- 最小路径下收益不明显

### 方案 C：改走替代站点或赛事方页面

做法：

- 优先抓赛事主办方站点、BO3、FACEIT 等页面
- 找不到时再兜底 HLTV

优点：

- 对 HLTV 单点依赖更低

缺点：

- 数据入口更分散
- “最近比赛 + 可下载 demo” 的统一发现逻辑更复杂
- 对当前最小目标不够直接

### 推荐

先采用方案 A。

原因：

- 当前最重要的是验证“能否拿到最近比赛并成功下载 demo”
- 方案 A 对结果最直接，能最快暴露真实障碍
- 如果之后需要更稳定的结构化解析，再在抓取层之上引入 `@beermonster/hltv`

## 4. 目标设计

### 4.1 范围边界

这轮只实现一个主进程原型链路，不接 UI，不改数据库 schema，不改现有 demo 导入主流程。

建议落点：

- `src/main/` 下新增一组 HLTV 原型文件
- 暂时通过单独 IPC 或开发用脚本入口触发

### 4.2 最小输出

一旦成功，原型返回以下信息：

- `matchId`
- `matchUrl`
- `eventName`
- `team1Name`
- `team2Name`
- `downloadedDemoPath`
- `downloadedFileSize`
- `source = "hltv"`

### 4.3 核心流程

1. 启动 Playwright 浏览器上下文
2. 打开 HLTV 最近结果页或最近比赛页
3. 发现最近一场可进入详情页的比赛
4. 打开比赛详情页
5. 判断该页是否存在 demo 下载入口
6. 若存在，则触发下载并等待文件落地
7. 校验文件真实存在、大小大于最小阈值、扩展名或 MIME 结果合理
8. 返回最小元数据与下载结果

### 4.4 页面抓取策略

首选页面：

- HLTV 结果页，因为“最近比赛”语义更明确

候选回退：

- HLTV 比赛页列表
- 已知具体比赛 URL

选择器策略：

- 不依赖单一深层 class 组合
- 优先抓比赛卡片链接、队伍名、赛事名、时间文本这类稳定信息
- 比赛详情页优先找包含 `download`、`demo`、`gotv` 等语义的链接或按钮

### 4.5 下载策略

- 使用 Playwright 原生 download 事件等待文件
- 下载目录指向系统临时目录下的独立工作目录
- 下载完成后立刻重命名到可读文件名
- 本轮不直接写入项目 `data/` 或正式 demo 库目录

建议命名：

- `hltv-<matchId>-<team1>-vs-<team2>.dem.zip`
- 若实际下载为 `.dem` 或其他扩展名，则保留实际后缀

### 4.6 成功判定

原型成功必须同时满足：

- 至少识别到 1 场最近比赛
- 成功打开其比赛详情页
- 成功触发并完成 1 个 demo 下载
- 落地文件存在且大小明显大于空文件

## 5. 组件拆分

建议最小拆分如下：

### `src/main/hltv-browser.js`

职责：

- 管理 Playwright 浏览器启动、上下文创建与关闭

### `src/main/hltv-discovery.js`

职责：

- 从最近结果页发现最近比赛链接
- 提取最小比赛元数据

### `src/main/hltv-demo-download.js`

职责：

- 在比赛详情页定位 demo 下载入口
- 执行下载并返回落地文件信息

### `src/main/hltv-prototype.js`

职责：

- 编排整个最小流程
- 返回统一结果对象

## 6. 失败模式与处理

### Cloudflare / Bot 阻断

表现：

- 页面打不开
- 被重定向到验证页
- 只有挑战页，没有真实 DOM

处理：

- 返回明确错误类型 `cloudflare_blocked`
- 记录 URL、页面标题和部分 HTML 片段用于调试

### 最近比赛没有 demo

表现：

- 比赛页可打开，但没有下载入口

处理：

- 继续尝试后续几场最近比赛
- 设置最大尝试数，避免无限扫描

### 下载按钮存在但下载失败

表现：

- 点击后没有 download 事件
- 下载文件为空
- 下载落地后扩展名或大小异常

处理：

- 标记为 `download_failed`
- 记录目标比赛 URL 与触发选择器

### 页面结构变化

表现：

- 结果页或比赛页无法定位关键元素

处理：

- 返回 `selector_mismatch`
- 输出抓取到的页面标题和调试片段

## 7. 验证方式

这轮只做最小验证：

1. 能否返回最近比赛元数据
2. 能否成功下载 1 个 demo 到临时目录
3. 能否对失败原因给出明确分类，而不是笼统报错

建议的最小测试方式：

- 先做 1 个端到端 smoke 脚本
- 再为“结果提取”和“下载结果归一化”补纯函数级测试

## 8. 后续扩展方向

如果最小路径打通，后续按以下顺序扩展：

1. 改造成主进程正式 IPC 能力
2. 接入 demo 库页面中的“从 HLTV 导入最近比赛”
3. 把 match metadata 与 demo 文件关联存入本地数据库
4. 评估是否引入 `@beermonster/hltv` 作为结构化解析层
5. 再考虑替代站点、兜底源和批量同步

## 9. 当前决策

- 当前不直接依赖普通 HTTP 抓 HLTV
- 当前不先接 UI 和数据库
- 当前先用 Playwright 验证最小抓取与下载链路
- 当前把 `@beermonster/hltv` 视为备选解析层，而不是最小路径的首要依赖
