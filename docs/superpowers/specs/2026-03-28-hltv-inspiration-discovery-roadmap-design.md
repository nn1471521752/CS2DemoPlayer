# HLTV 灵感发现与 Demo 分析全链路路线设计

- 日期：2026-03-28
- 项目：CS2DemoPlayer
- 状态：设计已确认，待写 implementation plan

## 1. 背景

当前项目已经落下了几条关键能力：

- `HLTV`：可浏览 recent matches、下载 archive、提取 `.dem`
- `Demo 库`：可管理本地 demo 并接入现有解析/回放链路
- `Replay`：已有 2D 回放、round 选择和基础标注入口
- `Entities`：已有候选池、审核流和正式战队/选手实体库第一版

但当前主链路仍然偏“工具拼接”：

- `HLTV` 更像 demo 下载入口，而不是灵感发现入口
- 用户能把比赛拉进本地，但还不能更快回答“这场值不值得看”
- 发现结果还不能稳定沉淀成后续分析队列和内容线索

项目长期目标仍保留 `demo -> 文稿 -> 视频`，但经本轮讨论后，当前阶段的主问题已经收敛为：

- 先更快找到值得看的比赛
- 再把它们送进 Demo 分析
- 最后再考虑内容生产闭环

这意味着项目的第一牵引能力不再只是“播放 demo”，而是“发现值得分析的 demo”。

## 2. 目标与非目标

### 目标

- 把产品主轴明确为 `灵感发现 > Demo 分析 > 内容生产`
- 把 `HLTV` 从“下载入口”升级为“灵感发现入口”
- 让第一阶段同时沉淀两类产物：
  - `待分析 demo 队列`
  - `灵感卡片池`
- 让 `HLTV -> 队列/卡片 -> Demo 分析` 形成最小闭环
- 为后续 `demo -> 文稿 -> 视频` 预留清晰连接点

### 非目标

- 本轮不把项目做成通用资讯平台或公开产品
- 本轮不先做重型内容编辑器、脚本编辑器或视频项目管理台
- 本轮不追求完全自动化的推荐系统
- 本轮不先解决所有回放 fidelity 和笔记体系问题
- 本轮不把 HLTV 之外的大量外部数据源一起接进来

## 3. 方案比较

### 方案 A：继续以 Demo 分析为主轴

做法：

- 继续优先补回放 fidelity、Entities 闭环和回合笔记体系
- `HLTV` 继续只承担 demo 获取入口

优点：

- 与现有仓库主线连续
- 不需要立刻定义“值得看”的判断系统

缺点：

- 无法优先解决“先看什么”的真实前置问题
- 项目仍会停留在“拿到 demo 后才有价值”
- `HLTV` 能力难以从入口升级为真正的产品牵引

### 方案 B：以 HLTV 灵感发现为第一主轴，推荐

做法：

- 把第一阶段定义为“更快找到值得看的比赛”
- 用 `规则辅助筛选 + 自动推荐` 驱动发现
- 发现结果沉淀为 `待分析 demo 队列` 与 `灵感卡片池`
- `Demo 分析` 作为下一跳承接这些结果

优点：

- 直接对应用户当前最关心的效率问题
- 能把 HLTV 从 demo 下载入口升级为选题/灵感入口
- 与长期 `demo -> 文稿 -> 视频` 目标一致，但不把项目一开始做重

缺点：

- 需要先定义一套“值得看”的信号体系
- 队列/卡片的结构要提前设计清楚

### 方案 C：直接做内容生产工作台

做法：

- 一开始就把 HLTV、demo、笔记、文稿和视频管理串成完整内容中台

优点：

- 终局完整
- 内容链路最直观

缺点：

- 范围过大
- 会把项目重心过早拖入内容生产细节
- 当前阶段难以聚焦“更快找到值得看的比赛”

### 结论

采用方案 B。

项目当前最合理的定位不是“通用播放器”，也不是“完整内容中台”，而是：

- 第一阶段先做 `HLTV 灵感发现工作台`
- 第二阶段把发现结果送进 `Demo 分析`
- 第三阶段再把分析结果接入内容生产

## 4. 产品定位

`CS2DemoPlayer` 的当前阶段定位调整为：

> 一个以 HLTV 为前置灵感来源、以 Demo 分析为第二落点、并为内容生产预留接口的桌面工作台。

这一定义有几个关键边界：

- 它优先服务“找到什么值得看”，而不是“所有比赛都要收进来”
- 它优先服务你自己的选题/分析工作流，而不是先做通用化产品能力
- `Demo 分析` 仍然重要，但不再是第一牵引入口

## 5. 宏观优先级

经本轮讨论，项目的宏观优先级固定为：

1. `灵感发现`
2. `Demo 分析`
3. `内容生产`

对应的产品逻辑是：

- 先解决“看什么”
- 再解决“怎么看”
- 最后解决“怎么产出”

## 6. 全链路结构

建议把后续全链路定义为：

1. `HLTV 浏览/筛选`
2. `发现值得看的比赛`
3. `沉淀为待分析 demo 队列`
4. `补充为灵感卡片`
5. `进入 Demo 分析`
6. `再衔接文稿/视频线索`

可压缩表达为：

`HLTV 浏览/筛选 -> 待分析 demo 队列 / 灵感卡片 -> Demo 分析 -> 内容生产`

其中：

- `待分析 demo 队列` 负责承接“下一步该看什么”
- `灵感卡片` 负责承接“为什么值得看”

这两者都应该是第一阶段的一等产物，而不是附属备注。

## 7. 第一阶段范围

### 7.1 第一阶段核心问题

第一阶段只优先回答一个问题：

> 如何更快找到值得看的比赛？

因此第一阶段要优化的是“发现效率”，不是“分析深度”。

### 7.2 第一阶段关键能力

第一阶段建议聚焦 4 类能力：

- `HLTV 浏览`
  - 更适合扫列表、比较比赛、快速判断是否值得点开
- `规则辅助筛选`
  - 先按比赛基础信息、结果特征、赛事强度等规则缩小范围
- `自动推荐`
  - 在规则筛完后，再把更值得优先看的比赛往前推
- `结果沉淀`
  - 把命中的比赛保存为 `待分析 demo 队列` 和 `灵感卡片`

### 7.3 第一阶段产物

第一阶段至少应形成以下产物：

- `筛选结果流`
- `推荐结果流`
- `待分析 demo 队列`
- `灵感卡片池`

### 7.4 第一阶段不做

- 不先做完整内容编辑/脚本编辑器
- 不先做完整视频项目管理
- 不先做特别复杂的知识库/标签体系
- 不先把推荐系统做成黑箱 AI

## 8. 第二阶段与第三阶段边界

### 8.1 第二阶段：Demo 分析

第二阶段负责消费第一阶段沉淀下来的队列和卡片。

重点包括：

- 回放可信度
- 回放交互和信息布局
- `Entities` 与 demo 级筛选入口闭环
- 围绕单场 demo 的分析与标注体验

这意味着 `Demo 分析` 不再孤立存在，而是变成灵感发现之后的下游动作。

### 8.2 第三阶段：内容生产

第三阶段才开始把分析结果更系统地转换为：

- 文稿线索
- 选题草案
- 视频候选

当前阶段只需要为这一步预留接口，不需要提前把产品做成内容中台。

## 9. 设计原则

后续路线建议遵守以下原则：

- `发现优先`
  - 先服务“找到值得看的比赛”，再服务“做深分析”
- `沉淀优先`
  - 发现结果必须能落入队列或卡片，不能只停留在浏览态
- `轻量判断优先`
  - 先给出足够好的筛选与排序，不追求一开始就做复杂推荐
- `链路完整优先`
  - 每一阶段都要能把结果交给下一阶段，而不是单点功能堆积
- `为自己而做`
  - 当前阶段优先服务你的真实工作流，不为泛化需求过度设计

## 10. 成功标准

如果这条路线成立，第一阶段至少应做到：

- 用户能比现在更快定位值得看的比赛
- 用户能把值得看的比赛稳定沉淀为 `待分析 demo 队列`
- 用户能为比赛补一张简短 `灵感卡片`
- 队列中的比赛能顺畅进入现有 demo 下载/导入/分析链路

项目层面则应体现为：

- `HLTV` 不再只是下载入口
- `Demo 分析` 不再是孤立工具
- `内容生产` 不再是脱节的远期口号，而是有上游输入的第三阶段

## 11. 后续实现顺序建议

建议按以下顺序推进：

1. 明确“值得看”的规则信号框架
2. 把 `HLTV` 页升级为灵感发现入口
3. 建立 `待分析 demo 队列`
4. 建立 `灵感卡片池`
5. 把队列项接回现有 Demo 分析链路
6. 再定义内容生产接口与落地方式

## 12. 当前仍待确认的问题

这份路线设计已经明确了宏观方向，但下列问题仍应留到下一轮细化：

- “值得看”的规则信号第一版具体由哪些字段组成
- `待分析 demo 队列` 的最小字段集合
- `灵感卡片` 的最小结构与编辑方式
- 推荐结果流与普通筛选结果流如何并存

这些问题不影响当前宏观路线成立，但会直接决定第一阶段的实现 plan。
