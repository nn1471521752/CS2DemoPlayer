# Entities 审核队列与全局实体库设计

- 日期：2026-03-23
- 项目：CS2DemoPlayer
- 状态：设计已确认，待写 implementation plan

## 1. 背景

当前项目已经有两类首页顶层能力：

- `Demo 库`：管理本地 demo 文件、解析状态与回放入口
- `HLTV`：发现近期比赛、下载 archive、提取 `.dem`

但项目还没有“全局实体层”。现有数据库和界面都以 `demo / round / tick` 为主，缺少下面两类面向筛选的长期实体：

- 战队
- 选手

用户希望后续能够围绕这些实体做筛选，但又不希望系统把解析结果直接写进正式实体库。当前已经明确的产品要求是：

- 全局实体库的第一版数据源以“已下载并成功解析的 demo”为主
- 不以 HLTV 为主数据源建立世界实体库
- 战队和选手应先进入待审核候选池
- 只有在用户批准后，才进入正式 `teams / players` 库
- 已有左侧导航需要保持干净，只保留页面列表
- 当前左侧底部详细 `Database` 信息栏应移除

这意味着本轮要做的不是简单加两张表，而是补一个完整的“候选 -> 审核 -> 正式入库 -> 浏览检索”的实体工作台。

## 2. 目标与非目标

### 目标

- 在首页左侧导航新增一个独立顶层页，用于管理全局实体
- 在该页内部提供 3 个 tab：
  - `待收录`
  - `战队`
  - `选手`
- 让系统从已下载并成功解析的 demo 中提炼候选战队和候选选手
- 候选实体必须经过用户批准后，才进入正式实体库
- 正式 `战队 / 选手` tab 支持基础搜索
- 为后续基于战队和选手的筛选能力打好数据库基础

### 非目标

- 本轮不把 HLTV 作为正式实体库的权威来源
- 本轮不自动把候选实体直接写入正式库
- 本轮不做战队详情页、选手详情页、转会历史或长期 roster 历史
- 本轮不做复杂别名合并工作流、人工编辑器或实体冲突解决台
- 本轮不把实体筛选直接接到 `Demo 库 / 回放页 / Notes` 的最终交互里
- 本轮不恢复或重做左侧 `Database` 详情为别的面板

## 3. 方案比较

### 方案 A：解析即入库

做法：

- 每次 demo 导入或重解析时，系统直接把识别到的战队和选手写入正式实体表

优点：

- 实现最短
- 数据即时可用

缺点：

- 容易把脏实体直接写进正式库
- 用户无法控制收录边界
- 后续清理和合并成本高

### 方案 B：候选池 + 人工批准入库，推荐

做法：

- 每次 demo 导入或重解析后，只更新候选池
- 用户在 `待收录` tab 中批准或忽略当前候选
- 只有批准动作才会把实体写入正式库

优点：

- 能把实体库和临时解析结果分层
- 用户对正式库有明确控制权
- 更符合“先审核，再入正式库”的产品要求

缺点：

- 比直接入库多一层工作流
- 需要额外的候选表和 UI

### 方案 C：HLTV 主库 + demo 校正

做法：

- 先从 HLTV 构建战队和选手世界库
- 再用本地 demo 解析结果补充或校正

优点：

- 世界库看起来更完整
- 后续接比赛详情可能更顺

缺点：

- 偏离当前“以已下载 demo 为主”的要求
- 命名和实体对齐成本高
- 会把本轮范围扩大到外部实体同步问题

### 结论

采用方案 B。

原因：

- 它同时满足“全局实体库”和“人工批准后再收录”这两个核心要求
- 它能在不重构现有回放链路的前提下，新增实体工作台
- 它给后续的筛选、tags、notes 留出了清晰边界

## 4. 页面与信息架构

### 4.1 左侧导航

左侧导航新增一个顶层页面入口，建议命名为 `Entities`。

导航原则：

- 左侧只保留干净的页面列表
- 移除当前左下角详细 `Database` 信息栏
- `Entities` 与 `Demo 库`、`HLTV` 同级

### 4.2 Entities 页整体结构

`Entities` 页作为一个独立首页内容页，顶部提供 3 个 tab：

- `待收录`
- `战队`
- `选手`

产品语义如下：

- `待收录`：审核工作台
- `战队`：正式战队实体库
- `选手`：正式选手实体库

这比把“待收录战队 / 待收录选手 / 已收录”做成 3 个 tab 更清晰，因为：

- 用户主操作流是“先审核，再进入正式库”
- 已收录后的战队和选手浏览、搜索方式不同
- 审核区内部完全可以再细分“候选战队”和“候选选手”

## 5. 待收录 tab 设计

### 5.1 结构

`待收录` tab 是一个审核工作台，而不是纯列表。

页面结构：

- 顶部 summary strip
- 下方两个 section
  - `候选战队`
  - `候选选手`

summary strip 至少显示：

- 候选战队数
- 候选选手数
- 涉及 demo 数
- 最近一次扫描时间

### 5.2 候选战队展示字段

每个候选战队至少展示：

- 候选名称
- 出现于多少个 demo
- 最近出现在哪个 demo
- 最近出现时间

第一版不强制要求展示完整 roster，也不强制展示对阵历史。

### 5.3 候选选手展示字段

每个候选选手至少展示：

- 选手名
- `steamid`
- 出现于多少个 demo
- 最近关联的候选战队
- 最近出现在哪个 demo
- 最近出现时间

### 5.4 审核动作

`待收录` 需要支持：

- 单项批准
- 单项忽略
- 多选后批量批准
- 多选后批量忽略

`忽略` 的语义已经确认：

- 只忽略当前这次候选记录
- 未来如果在新的 demo 中再次出现，或者候选证据发生变化，可以重新回到 `待收录`

因此本轮不做永久黑名单。

## 6. 正式实体库 tab 设计

### 6.1 战队 tab

`战队` tab 用于展示已经正式收录的战队库。

每行至少显示：

- 战队名
- 关联 demo 数
- 关联选手数
- 最近出现时间

顶部提供搜索框，用于按战队名快速过滤。

### 6.2 选手 tab

`选手` tab 用于展示已经正式收录的选手库。

每行至少显示：

- 选手名
- `steamid`
- 最近关联战队
- 关联 demo 数
- 最近出现时间

顶部提供搜索框，用于按选手名或 `steamid` 快速过滤。

## 7. 数据口径

### 7.1 数据源

第一版正式实体库与候选池都只以“已下载并成功解析的 demo”为数据源。

这意味着：

- 可以消费现有 parser 和数据库已落下来的事实数据
- 不需要先建立 HLTV 世界实体库
- HLTV 当前最多只作为 demo 获取入口，不作为实体收录权威来源

### 7.2 战队候选来源

战队候选来自现有回放链路中的稳定队名字段：

- `team_display`
- `team_clan_name`

只有能稳定识别出真实队名的候选，才进入待收录池。

### 7.3 选手候选来源

选手候选只收录带稳定身份标识的选手。

第一版规则：

- 只收带 `steamid` 的选手
- 没有 `steamid`、只有名字的记录不进入全局 `players` 候选池

这样可以降低重名、改名、同人多条的风险。

## 8. 数据模型设计

### 8.1 正式实体表

建议新增正式实体表：

- `teams`
- `players`

主键策略：

- `players`：以 `steamid` 作为内部主键
- `teams`：以标准化后的队名字符串作为第一版内部主键

同时保留未来扩展位，例如：

- 外部来源 id
- 展示名
- 标准化名
- 收录时间
- 最近出现时间

### 8.2 候选表

建议新增候选表：

- `team_candidates`
- `player_candidates`

候选表负责存：

- 当前候选实体的展示信息
- 证据汇总信息
- 状态信息
- 最近一次扫描时间

候选状态至少需要区分：

- `pending`
- `approved`
- `ignored`

其中 `ignored` 不是永久封禁，只表示当前批次候选已被跳过。

### 8.3 关联表

为了支持后续筛选，建议同时准备实体与 demo 的关联表：

- `team_demo_links`
- `player_demo_links`

它们至少需要记录：

- 实体主键
- demo checksum
- 首次出现时间
- 最近出现时间

这样后续做“按战队筛 demo”“按选手筛 demo”时，不需要每次回扫所有 tick 事实表。

## 9. 更新时机

更新分两层：

### 9.1 自动更新候选池

每次发生以下动作后，自动更新候选池：

- 新 demo 导入并完成解析
- 已有 demo 重新解析

自动更新的对象是候选池，而不是正式实体库。

### 9.2 手动更新正式实体库

正式 `teams / players` 只有在用户明确执行以下动作时才变化：

- 批准单个候选
- 批量批准多个候选

忽略动作不会把实体写进正式库。

## 10. 交互与状态边界

### 10.1 空状态

`Entities` 页需要有明确空状态：

- 没有任何候选时，`待收录` 明确显示“还没有待审核实体”
- 没有已收录实体时，`战队 / 选手` tab 明确显示空库提示

### 10.2 重复候选

候选池不应该因为同一实体在多个 demo 中出现就重复生成多条相同记录。

候选项应该是“按实体聚合后的候选”，并带汇总证据：

- 出现过的 demo 数
- 最近出现 demo
- 最近出现时间

### 10.3 批量操作失败

如果批量批准或批量忽略时有部分失败：

- 已成功的项应保留成功结果
- 失败项应留在当前列表并给出错误提示
- 不要求整批事务性回滚

## 11. 对现有系统的影响

本轮新增的是“实体层”和“审核层”，不应破坏现有三条主线：

- demo 导入与解析
- 回放与 round 选择
- HLTV 下载与 demo 提取

对现有首页的主要影响只有两点：

- 左侧导航新增 `Entities`
- 左下角 `Database` 详情栏移除

## 12. 验证标准

设计完成后的实现应至少满足：

- 左侧导航存在独立 `Entities` 页面入口
- 左侧导航不再显示详细 `Database` 信息栏
- `Entities` 页有 `待收录 / 战队 / 选手` 三个 tab
- 从已解析 demo 中能生成候选战队和候选选手
- 用户批准后，候选可进入正式 `teams / players`
- 用户忽略后，当前候选消失，但未来可重新出现
- `战队 / 选手` tab 都支持基础搜索

## 13. 后续实现顺序建议

建议按以下顺序实现：

1. 数据库 schema：正式实体表、候选表、关联表
2. 候选提炼与聚合逻辑：从现有 demo 解析结果生成候选
3. 审核动作：批准、忽略、批量操作
4. `Entities` 页与 3 个 tab
5. `战队 / 选手` tab 搜索

这个顺序可以让 UI 和数据逻辑逐步接起来，同时保证已有链路风险最小。
